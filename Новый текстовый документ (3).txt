Модуль № 1
Разработка базы данных средствами СУБД
Продолжительность выполнения: 0 ч. 50 мин.

Задание
Компания «Мир настольных игр» занимается продажей настольных игр, пазлов и аксессуаров.
Очень важно постоянно получать актуальную информацию об остатках товаров на складах, поэтому необходимо реализовать следующие модули.

При запуске приложения окно входа – первое, что видит пользователь. На ней пользователю предлагается ввести свой логин и пароль, взятый из базы данных, или есть возможность перейти на экран просмотра товаров (без фильтрации, сортировки, поиска) в роли гостя.

Только после удачной авторизации пользователь получает доступ к остальным модулям системы:

Авторизованный клиент может просматривать товары (без фильтрации, сортировки, поиска);

Менеджер может просматривать товары (с фильтрацией, сортировкой, поиском), просматривать заказы;

Администратор может просматривать (с фильтрацией, сортировкой, поиском) / добавлять / редактировать / удалять товары, просматривать / добавлять / редактировать / удалять заказы.

На основе описания предметной области необходимо создать базу данных в выбранной СУБД для разрабатываемой системы.
Обязательна 3 нормальная форма с обеспечением ссылочной целостности. При разработке базы данных обратите внимание на согласованную схему именования, создайте необходимые первичные и внешние ключи.

Получить ER-диаграмму средствами СУБД или ПО для построения диаграмм (UML): ER-диаграмма должна быть представлена в формате PDF и содержать таблицы, связи между ними, атрибуты и ключи (типами данных на данном этапе можно пренебречь).

Заказчик системы предоставил файлы с данными (с пометкой import в ресурсах) для переноса в новую систему. Необходимо подготовить данные файлов для импорта и загрузить в разработанную базу данных.

Сохранить полученные результаты: создать скрипт БД (или файл конфигурации с данными .dt для платформы 1С).

Модуль № 2
Разработка алгоритма и создание приложения
Продолжительность выполнения: 0 ч. 40 мин.

Задание
Сформировать алгоритм разработки приложения: оформить алгоритм в виде блок-схемы согласно стандарту ГОСТ 19.701-90. Документ представить в формате .pdf.

Компоненты системы должны иметь единый согласованный внешний вид, соответствующий руководству по стилю. Заголовок окна (страницы) должен соответствовать назначению. Следует установить иконку приложения (если это реализуемо) и логотип компании на главной форме из ресурсов.

Оформление кода:

идентификаторы должны отражать их назначение и соответствовать CamelCase (для C#, Java), snake_case (для Python) или стандарту 1С;

допустимо использование не более одной команды в строке.

Созданную базу данных необходимо подключить к приложению, реализующему необходимый функционал. Список товаров должен отображать информацию из базы данных.

Авторизация
При запуске приложения окно входа – первое, что видит пользователь. На ней пользователю предлагается ввести свой логин и пароль, взятый из базы данных, или есть возможность перейти на экран просмотра товаров в роли гостя.

Только после удачной авторизации пользователь получает доступ к остальным модулям системы.

Реализуйте необходимые интерфейсы для всех пользователей системы (гость, авторизованный клиент, менеджер, администратор).
После входа в любую учетную запись должна быть реализована возможность выхода на главный экран – окно входа.
При переходе в любую учетную запись в интерфейсе (правый верхний угол) должны отображаться ФИО пользователя.

Список товаров (каталог)
Очень важно постоянно получать актуальную информацию об остатках товаров на складах, поэтому необходимо реализовать вывод товаров, которые хранятся в базе данных:

Элемент	Описание
Фото товара	При отсутствии изображения – картинка‑заглушка picture.png
Наименование товара	Название настольной игры / пазла / аксессуара
Категория	Например: «Стратегии», «Детские игры», «Пазлы», «Аксессуары»
Описание	Краткое описание (количество игроков, возраст, время партии)
Производитель	Hasbro, Hobby World, Ravensburger, Zvezda и др.
Поставщик	Название компании‑поставщика
Цена	В рублях (с копейками)
Единица измерения	шт.
Количество на складе	Целое число
Действующая скидка	% от 0 до 100
Подсветка строк
Если товара нет на складе (количество = 0) – строка выделяется серым цветом.

Если скидка превышает 30% – фон строки становится #FFD966 (жёлто-золотистый).

Если на товар действует скидка (любая >0), то основная цена должна быть перечёркнута красным цветом, а рядом указана итоговая цена чёрным цветом.

Выполнить отладку модуля для проверки функциональности: приложение должно корректно работать, не должно происходить аварийного завершения.
Создайте документ .docx и вложите туда скриншоты корректной работы системы.

Модуль № 3
Реализация сортировки, фильтрации, поиска и CRUD для товаров
Продолжительность выполнения: 1 ч. 30 мин.

Задание
Реализовать последовательный пользовательский интерфейс, позволяющий перемещаться между существующими окнами (страницами) в приложении (в том числе обратно, например, с помощью кнопки «Назад»). Обеспечить соответствующий заголовок на каждом окне.

Реализовать обработку исключительных ситуаций:

уведомлять пользователя о совершаемых ошибках или запрещённых действиях;

предупреждать о необратимых операциях;

окна сообщений (ошибка, предупреждение, информация) должны иметь соответствующий заголовок и пиктограмму;

текст сообщения должен быть полезным, содержать информацию об ошибке и порядок действий для её исправления;

использовать визуальные подсказки при вводе данных.

Комментарии в коде должны присутствовать только в местах, требующих дополнительного пояснения.

Поиск, фильтрация, сортировка (только для менеджера и администратора)
Поиск – по всем текстовым полям (наименование, описание, категория, производитель, поставщик) одновременно.

Сортировка – по цене и по количеству на складе (по возрастанию и убыванию).

Фильтрация – по диапазону скидки:

0 – 9,99%

10 – 19,99%

20% и более

Первый элемент выпадающего списка – «Все диапазоны» (сброс фильтра).

Важно:
Поиск, сортировка и фильтрация должны происходить в реальном времени (без нажатия кнопки «Найти»/«Отфильтровать»).
Фильтрация и поиск применяются совместно. Параметры сортировки, выбранные ранее пользователем, сохраняются при фильтрации и поиске.

CRUD товаров (только администратор)
Добавление и редактирование – в новой форме (окне). Переходы:

на редактирование – при нажатии на строку товара в списке;

на добавление – по кнопке «Добавить товар».

Поля формы:
фото товара (предпросмотр), наименование, категория (выпадающий список), описание, производитель, поставщик (выпадающий список), цена, единица измерения, количество на складе, скидка.

Цена – неотрицательная, может включать сотые части.

Количество – неотрицательное целое.

ID товара – при добавлении не отображается (вычисляется max+1), при редактировании – только для чтения.

Изображение

Загружается администратором.

Сохраняется в папку приложения.

В БД хранится путь к файлу.

При замене – старое изображение удаляется с диска.

Ограничение размера фото: 300×200 пикселей.

Запрет на открытие более одного окна редактирования (одновременно).

Удаление товара

Товар, который присутствует в заказах, удалить нельзя.

После любого изменения (добавление/редактирование/удаление) список товаров обновляется с сохранением текущих параметров поиска/фильтрации/сортировки.

Модуль № 4
Добавление функционала работы с заказами для Администратора и Менеджера
Продолжительность выполнения: 1 ч. 00 мин.

Задание
В интерфейсах Менеджера и Администратора добавить кнопку «Заказы» для вывода информации по заказам согласно макету.

Требуемые возможности:

Просмотр списка заказов (артикул заказа, статус, адрес пункта выдачи, дата заказа, дата выдачи, ФИО клиента, итоговая сумма).

Добавление и редактирование заказов – только для администратора.
Форма добавления/редактирования содержит поля:

Артикул заказа (уникальный)

Статус заказа (выпадающий список: «Новый», «В обработке», «Завершён»)

Адрес пункта выдачи (список из справочника)

Дата заказа

Дата выдачи (может быть пустой)

Удаление заказа – только администратор.

После сохранения или удаления – обновление списка заказов.

Переходы:

на редактирование – по нажатию на строку заказа;

на добавление – по кнопке «Добавить заказ».

Практические результаты (итоговые файлы)
Все практические результаты должны быть переданы путём загрузки файлов на предоставленный репозиторий системы контроля версий:

Исходный код приложения (структура папок с файлами, не архивом).

Исполняемые файлы (собранное приложение).

Файл скрипта базы данных (CREATE/INSERT).

Прочие графические/текстовые файлы (ER-диаграмма, блок-схема, скриншоты, документация).
